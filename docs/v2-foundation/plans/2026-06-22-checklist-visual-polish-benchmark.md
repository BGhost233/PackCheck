# 核查清单页 UI 质感提升计划 — 对标便单

> 状态：方案待评审（开发铁律 §0：先出方案，等确认再动手）
> 范围：行程详情页的「带格子的核查清单」整页质感（收缩态 / 展开态 / 拖动态 + 顶部信息区 / 进度条 / 添加入口 / 空态）
> 不动范式：严格遵守 vision v3 §4「格子即清单，一体不分家」与 §4.4.1 空/满展示规则。本计划只做视觉与交互反馈层的精装修，不触碰信息架构。
> 沉淀日期：2026-06-22

---

## 0. 一句话定调

便单和我们是**同构产品**（格子 + 列表项 + 勾选），它看起来「成立」、我们看起来「半成品」，差距不在功能，在三件事：**容器存在感、状态对比度、交互物理感**。本计划把这三件事补齐，并守住山野绿单主题的克制基调——拿到便单的「容器感」，但不抄它的「整块染色」。

---

## 1. 现状诊断（基于截图 + 代码双重核对）

### 1.1 问题一：容器存在感缺失 —— 简陋感的根因

便单收缩态每个象限是**完整的染色实体盒子**（淡色填充 + 同色描边 + 圆角），哪怕只有一两项也视觉闭合、有重量。

我们的现状（`ZoneGridCell.ets`）：

- **空格子**：`borderRadius(CARD_RADIUS)` + `1.5px DIVIDER_COLOR` 虚线框，撑满 `cellHeight`（首屏反算 ~148vp）。符合 vision §4.4.1，但虚线偏重、面积偏大。
- **有内容格子**：内层白卡 `CARD_BG` + `SHADOW_SUBTLE`（`#0A000000`，几乎不可见）。阴影太弱 → 白卡像「直接印在背景上的文字」，没有「装在盒子里」的感觉。
- **格子外层**：`border.width = 0`（仅拖拽高亮时才 2.5px 绿框）。即静态态**没有任何容器边界**，全靠那道极弱阴影撑场。
- **标题区分**：仅 `4×16` 的 Zone 色条 + 同色图标 + 同色文字（`buildTitleRow`）。色彩只点在标题，**容器本体无色彩锚点**。

> 根因：我们把「格子」做成了「分组标题 + 飘着的内容」，便单把「格子」做成了「物理容器」。容器感缺失 = 简陋感来源。

### 1.2 问题二：空/满视觉权重倒置 —— 收缩态区分不清的根因

vision §4.4.1 的设计意图是对的（「有的鼓起来、没的扁着」），但实机效果是反的：

- 空格子 = 一大块 ~148vp 的虚线灰方块 → 视觉重量**很重**；
- 满格子（只 1 件装备时）= 白卡 + 极弱阴影 → 视觉重量**很轻**。

结果：空的比满的还抢眼，层级倒置。用户扫一眼注意力被四个大灰方块吸走，反而看不清「我已经装了哪些」。便单的空象限因为**有底色填充**，空白也是「实心盒子」，不违和；我们选了不填充（见 §3 决策），空白就是真空白，必须靠**压缩高度 + 极致淡化**来压住。

### 1.3 问题三：拖动态「全页变灰」—— 实为实现遗留，非设计意图

**关键纠正**（代码核对，`UnifiedChecklistView.ets`）：代码里**没有**「拖动时全页变灰」的设计。截图里的灰，是长按浮层 `buildLongPressOverlay` 的**菜单态蒙层** `#33000000`（20% 黑）。交互时序是：

1. 长按 → 弹「编辑 / 移动 / 移除」菜单（带蒙层，全页压灰，合理）
2. 从菜单态直接滑动 → 进入 `dragging` 阶段（绿框 + 跟手小药丸出现）
3. **但蒙层 opacity 没在进入拖拽时降下来** → 拖拽全程背景还压着那层灰

截图正是这个尴尬中间态：菜单蒙层未退 + 已进拖拽。这不是「拖动设计成变灰」，是 **`overlayPhase` 从 `menu` 转 `dragging` 时蒙层未淡出**的遗留。修起来比「重写拖动交互」轻得多——只需让蒙层在 dragging 阶段 opacity → 0。

附带的拖动质感问题：

- 跟手胶囊（`buildLongPressOverlay` 阶段2）只有 `CARD_BG` + 阴影，**无浮起缩放、无轻微倾斜**，不像「被拎起来」。
- 源位置（被拖项原格）拖动时**无占位反馈**（不变虚位），用户不知道「从哪拿的」。
- 拖拽时只有目标格子高亮，源格子无「掏空」暗示。

### 1.4 问题四：展开态丢失容器 —— 同一个病根

`FocusedZoneView.ets` 展开后：顶栏（× + Zone 名 + 进度）+ 全量列表 + 底部虚线添加行，**下方大片 PAGE_BG 白底裸露**。便单展开后象限色块仍在、始终「有形」；我们展开后容器消失，比收缩态还空旷。

### 1.5 问题五：两列高度错落

`Grid + columnsTemplate('1fr 1fr')` 下，左右两列各自高度由内容决定 → 满格子和空格子并排时高度不齐、参差错落（截图明显）。便单四象限是严整 2×2 矩阵，整齐感是「精致」的基础。

## 2. 差距总表

| 维度 | 便单 | 我们（现状） | 病根 |
|---|---|---|---|
| 容器 | 染色实体盒子 | 标题点色 + 飘着的内容 + 极弱阴影 | 容器感缺失 |
| 空/满对比 | 自然分层（空也是实心盒） | 空的比满的还重，层级倒置 | 视觉权重错配 |
| 拖动反馈 | 浮起 + 目标高亮，页面常亮 | 菜单蒙层未退 → 拖拽全程压灰 | 实现遗留（非设计） |
| 展开态 | 容器放大、始终有形 | 容器消失、大片空白 | 同样的容器缺失 |
| 网格对齐 | 严整 2×2 矩阵 | 两列高度参差错落 | 列高未对齐 |
| 装饰密度 | 徽标 / 分隔线 / 勾选态丰富 | 几乎裸 | 信息装饰不足 |

## 3. 已对齐的设计决策（评审锁定）

| 决策项 | 选择 | 说明 |
|---|---|---|
| **容器风格** | 克制白卡 + 柔和阴影 + **极淡同部位色描边** | 不抄便单整块染色。白卡拿容器感，描边用 `ZONE_*_TINT`（已存在的 20% 部位色）做识别，守住山野绿单主题克制基调 |
| **空格子** | 压缩到 ~120-140vp 呼吸位 + 极淡虚线 + 中心极淡提示文案 | 既守 vision §4.3「空格子始终可见做提醒」，又不让大片空白压过白卡（避免 §1.2 层级倒置） |
| **拖动反馈** | 全程常亮：被拖项浮起（scale + 阴影 + 轻微倾斜）+ 目标格高亮 + 源位虚线空槽 | 彻底干掉拖拽态压灰（修 §1.3 遗留），不引入任何 dim |
| **布局** | 保持两列，但改矩阵对齐（修列高参差） | 不改 2 列信息架构，只把高度对齐 |
| **范围** | 整页质感提升（含顶部信息区 / 进度条 / 添加入口 / 空态） | 不止格子 |

### 3.1 关键资产复用（降本）

代码核对发现**部位色描边几乎零成本**：`Colors.ets` 已存在全套 `ZONE_*_TINT`（`#33` 前缀 = 20% 部位色）：

```
ZONE_HEAD_TINT  = '#3342A5F5'   ZONE_UPPER_TINT = '#33EF5350'
ZONE_LOWER_TINT = '#335C6BC0'   ZONE_FEET_TINT  = '#338D6E63'
ZONE_CARRY_TINT = '#33FFA726'   ZONE_SLEEP_TINT = '#337E57C2'
ZONE_MISC_TINT  = '#3378909C'
```

20% 部位色直接做描边偏跳，需新增更淡的 `ZONE_*_STROKE` 一组 token。这是本计划唯一需要新增的 token。

**描边透明度与饱和度挂钩（评审采纳 D-1）**：不用全部固定同一透明度，而是按 Zone 色饱和度差异微调，让七个描边的「感知存在感」均衡（而非「物理透明度」一致）：

| Zone | 色值 | 饱和度 | 描边透明度 |
|---|---|---|---|
| 头部 Head | `#42A5F5` 蓝 | 高 | ~8%（`#14`） |
| 上身 Upper | `#EF5350` 红 | 高 | ~8%（`#14`） |
| 下身 Lower | `#5C6BC0` 靖蓝 | 高 | ~8%（`#14`） |
| 携带 Carry | `#FFA726` 橙 | 中 | ~11%（`#1C`） |
| 睡系 Sleep | `#7E57C2` 紫 | 中 | ~11%（`#1C`） |
| 足部 Feet | `#8D6E63` 棕 | 低 | ~14%（`#24`） |
| 杂项 Misc | `#78909C` 灰蓝 | 低 | ~14%（`#24`） |

> 上表透明度为初始建议值，以真机并排时「七色描边看起来是否一样深」为准微调。

## 4. 详细优化方案（逐组件）

### 4.1 收缩态满格子：白卡容器化（`ZoneGridCell.buildContentCell`）

**目标**：让满格子成为有边界、有重量的实体盒子。

| 项 | 现状 | 目标 |
|---|---|---|
| 阴影 | `SHADOW_SUBTLE #0A000000, radius 8, offsetY 2` | 升一档：`radius 12, color #14000000(SHADOW_LIGHT), offsetY 4`——刚好「浮起一点点」，不重 |
| 描边 | 无 | `0.5-1px` `ZONE_*_STROKE`（10-12% 部位色）细描边，给容器一个有颜色的边界 |
| 圆角 | `CARD_RADIUS`（20vp） | 保持 |
| 标题行 | `4×16` 竖色条 + 同色图标文字 | **整条标题行浅染 `ZONE_*_TINT` 背景 + `borderRadius(8)`**（评审锁定 D-3）。关键：**色条/图标/名称保 `TEXT_MAIN` 深色不跟着染部位色**，保证可读性。浅彩底 + 深色字 = 便单「标签头」感，且与 §4.7 展开态顶栏共用同一色彩语言（转场时标题区连续） |
| 内边距 | 现 `ChecklistRow padding left/right 12` | 卡内统一 `padding 12-14`。**收缩态不加行间分隔线**（评审采纳 S1）——每格最多 4 件，加分隔线会把「一个盒子」碎成「4 个独立条」，与「容器是一个盒子」目标相悛；收缩态靠行高留白分隔，分隔线仅用于展开态全量列表（见 §4.5） |

**勾选态丰富化**（对标便单的 checkbox 完成态）：已勾选项 → 圆圈填充部位色 + 白勾 + 文字 `TEXT_TERTIARY` + 删除线（弱化已完成），未勾选 → 空心圈。当前看图勾选态偏弱，需要加强「打勾后的成就反馈」。

### 4.2 收缩态空格子：压缩 + 极致淡化（`ZoneGridCell.buildEmptyCell`）

**目标**：守 vision §4.3「始终可见」，但把视觉重量压到最低，修正层级倒置。

| 项 | 现状 | 目标 |
|---|---|---|
| 高度 | 撑满 cellHeight（~148vp） | **`min(cellHeight, 132vp)`**（评审采纳 D-2）——语义：空格子最多 132vp，但若满格子因密度反算比这矮，空格子跟着矮。**始终同行等高**（不破对齐），只是空格子不会撑满 |
| 虚线 | `1.5px DIVIDER_COLOR #F0F0F0` | 降到 `1px` + 更淡（`TEXT_HINT` 级，或 `ZONE_*_STROKE` 的更淡变体，让空格子也带一丝部位色暗示但几乎不可见） |
| 「+」图标 | `fontSize 20, TEXT_TERTIARY` | 降到 `18, PLACEHOLDER_COLOR #BBBBBB` |
| 提示文案 | 无 | 「+」下方加一行极淡文案，如「头部」（部位名）或「添加头部装备」，`FONT_SIZE_CAPTION, PLACEHOLDER_COLOR`——满足 §1.2 vision「空格子是提醒」 |

> 实现注意：空格子高度 = `min(cellHeight, 132)`，与 §4.3 列高对齐策略联动——同行两格取等高，空格子以 132vp 封顶且不超过同行高度。

### 4.3 网格对齐：修两列参差（`UnifiedChecklistView.buildGridState`）

**目标**：从「参差错落」变「严整矩阵」。

- 现 `Grid 1fr 1fr` 各 GridItem 高度由 `cellHeight` 统一传入，但满格子内容白卡 `layoutWeight(1)` + 空格子虚线区 `layoutWeight(1)`，理论应等高——若实机仍参差，需排查 `cellHeight` 是否对所有 cell 一致下发。
- 决策（评审采纳 D-2）：**同行两格取等高 = 该行满格子高度（按 cellHeight）**；空格子高 = `min(同行高, 132vp)`。即：满格子决定行高，空格子在该行高内「不超 132vp」地居中留白（空白区在虚线框外的 cell 内垂直居中）。这样同行基线对齐，又不让空格子视觉压过满格子。若整行都是空格子，两格同为 132vp。
- 杂项格跨两列（现 `columnStart 0 / columnEnd 1`）保留。

### 4.4 拖动态：去灰 + 浮起 + 源位空槽（`UnifiedChecklistView` + `ZoneGridCell`）

**目标**：页面全程常亮，被拖项有「被拎起」的物理感。

**4.4.1 修菜单蒙层未退（核心修复，最高优先级）**

`buildLongPressOverlay` 的蒙层 `Column().backgroundColor('#33000000').opacity(this.overlayVisible ? 1 : 0)`——改为：

```
.opacity(this.overlayPhase === 'dragging' ? 0 : (this.overlayVisible ? 1 : 0))
```

即进入 `dragging` 阶段蒙层 opacity → 0（带 `EASE_FADE` 过渡），页面恢复常亮。这一改直接解决「拖动全页变灰」。

**4.4.2 跟手胶囊浮起物理感**（`buildLongPressOverlay` 阶段2 胶囊）

| 项 | 现状 | 目标 |
|---|---|---|
| 缩放 | 无 | 进入 dragging 时 `scale 1.05`（拎起略放大） |
| 阴影 | `radius 16, OVERLAY_DIM, offsetY 6` | 升 `radius 20, #28000000(SHADOW_MEDIUM), offsetY 8`（离地更高） |
| 倾斜 | 无 | 轻微 `rotate 2-3°`（可选，便单同款，给「被手指拎着晃」的灵动感） |
| 进入动画 | opacity 切换 | 长按转拖拽瞬间用 `SPRING_PRESS` 弹一下 scale，强化「拎起」 |

**4.4.3 源位置虚线空槽**（`ZoneGridCell` 满格子内被拖项）

被拖项在源格内的原行 → 拖动期间变为**虚线占位空槽**（不消失、不塌陷），告诉用户「从这里拿走的，松手放回会回这」。需要把「当前被拖 itemId」透传到 `ZoneGridCell`，命中行渲染为虚线空位。

**4.4.4 目标格高亮优化**（`ZoneGridCell` 现有 `isDropTarget`）

现状已有（`2.5px PRIMARY_COLOR` 绿框 + `#402D7D46` glow），保留。可微调：高亮时格子背景叠一层极淡 `LIGHT_PRIMARY_COLOR #E8F5E9`，让「可放这里」更明确。

### 4.5 展开态：找回容器（`FocusedZoneView`）

**目标**：展开后仍「有形」，不再大片白底裸露。

- 全量列表外**包一层白卡容器**（即 §4.7 的 `buildZoneShell` 外壳：`CARD_BG` + `SHADOW_LIGHT/MEDIUM` + `CARD_RADIUS` + `ZONE_*_STROKE` 描边），呼应收缩态满格子的视觉语言（收→展是同一个盒子放大，符合 geometryTransition 一镜到底心智）。
- 顶栏 Zone 名区域浅染 `ZONE_*_TINT` 背景（色条/图标/名保 `TEXT_MAIN`），与收缩态标题行 §4.1 保持完全一致——这是转场两端外观连续的前提。
- **分隔线仅用于此处（展开态全量列表）**（评审 S1）：展开态条目多，`0.5px DIVIDER_COLOR` 行间分隔线提升可扫读性；收缩态最多 4 件不加分隔线（见 §4.1）。
- 底部虚线「添加到 XX」按钮保留，但收进白卡容器内（作为列表末项），不裸露在白底上。
- 列表内容少时，白卡容器维持一个最小高度，内部留白在**容器内**（学便单：盒子在，空白也在盒子里，不空旷）。

### 4.6 顶部信息区 + 进度条质感（`TripDetailPage` 顶部 + `LoadoutProgressBar`）

**目标**：顶部「非洲 / 2026年7月1日·乞力马扎罗·海拔5895m / 已装包 0/5 0%」目前偏朴素，提升到「行程档案」的精致感。

- **进度条**（`LoadoutProgressBar`）：**保持细条，克制设计**（评审采纳 S2）——核心价值是「一眼看完成度」，花哨渐变反而延长认知时间。只加 `SPRING_COUNTER` 数字滚动 + 100% 时主色高亮（庆祝功能层已有）。颜色用 `PRIMARY_COLOR` 纯色，0% 态用 `PROGRESS_INACTIVE`，不做多段渐变。
- **行程元信息行**（日期·地点·海拔）：**图标克制**（评审采纳 S3）——只给海拔（三角图标）加图标点睛，日期/地点保持纯文字。元信息行是紧凑摘要不是图标阵列。数字用 `tnum` 等宽 + `AMBER_ACCENT` 点睛（呼应 vision §3「数据是钥匙」的留白数据叙事）。
- **整体**：顶部信息区与格子区之间用 §3.4 的呼吸间距，不要挤。

---

## 4.7 收→展转场重构：从「翻页」到「格子原位放大」（对标便单核心诉求）

> 用户原话：「便单点格子后边框自然放大、再点又自然缩小回原位，无缝共享元素转场，整体都以格子变大变小划分空间。我们现在点完像进入另一个页面，和之前的边框没有联动。」

### 4.7.1 诊断：geometryTransition 已配，为何退化成「翻页」

代码里两端**确实配了 `geometryTransition('zone_' + zone)`**（网格端 `UnifiedChecklistView` GridItem L484 / 聚焦端 `FocusedZoneView` L213，同一 id），骨架是对的。但有三个硬伤让它退化成「淡出淡入两套界面」：

**硬伤一：两端外观完全不同 → 没有「边框连续放大」可言。**
`geometryTransition` 只对**几何位置/尺寸做插值**，不会把 A 的外观平滑变成 B 的外观。现状：
- 网格端 `ZoneGridCell` = 白卡 + 标题色条 + 4 件预览 + 圆角 20vp + （Phase B 后）描边
- 聚焦端 `FocusedZoneView` = `backgroundColor(PAGE_BG)` **整屏羽白底**，无卡片、无圆角、无边框

转场时系统只能「缩放几何框 + 交叉淡入淡出内容」。用户看到的是「小白卡淡出 + 大白页淡入」，中间**没有任何边框在连续放大**——因为聚焦端根本没有边框。这是「像进入另一个页面」的直接主因。

> 便单能做到「边框自然放大」，前提是**它的展开态本身就是一个放大的格子**（仍带边框、圆角、底色），所以收→展时那个边框是同一个、连续存在的，只是变大。

**硬伤二：聚焦态铺满全屏 → 失去「格子在空间里」的尺度感。**
`FocusedZoneView` 是 `width('100%') height('100%')` 整屏。便单展开态**不铺满**，放大后仍是有边界的盒子浮在原位附近，四周能看到其他格子余光，空间始终由格子划分。一铺满，空间感就断了，自然像翻页。

**硬伤三：背景其他格子硬遮消失 → 无联动感。**
展开时其他 6 格靠 `opacity(focusedZone == zone ? 0 : ...)` 控制（L485），但只有被点格参与 geometryTransition，其余格被整屏 `PAGE_BG` 硬遮挡住，没有「被推开 / 虚化」的联动，强化了「换页」错觉。

### 4.7.2 核心思路（一句话）

**让聚焦态在视觉上 = 收缩态格子的等比放大版**——收缩态和展开态共用同一套外壳视觉（白卡 + 同色描边 + 圆角 + 标题色条），转场时外壳几何连续插值、内部内容交叉淡入。这样中间帧才是「同一个边框在放大」，而非「两套界面互相淡出淡入」。

### 4.7.3 已锁定决策（评审确认）

| 决策项 | 选择 |
|---|---|
| 展开尺寸 | **放大但不铺满全屏**：保留卡片边框/圆角，四周留边距（如左右各 20-24vp、上下避开顶部信息区与底部安全区），能看到其他格子余光 |
| 外壳连续性 | **两端共用一套外壳视觉**，展开态重做成「`ZoneGridCell` 外壳的放大版」（这是做到便单效果的唯一正解） |
| 目标 Rect 锚定 | **就地放大、几何中心连续**（评审采纳 R1）——目标 Rect 的中心**锚定原格子的几何中心**，不强行居中到屏幕中央。若点的是偏下行，让卡片在原位附近就地长大（中心尽量不动），这才是便单「原位放大」的本质；强行「跳到屏幕中央再放大」恰恰会制造新的跳变感 |
| 背景联动 | **全屏半透明蒙层压暗**（评审采纳 R1，从「缩放网格」降级）——压暗用覆盖全屏的蒙层，不依赖网格是否在视口；网格层只做**轻微下沉 + 虚化**，**不做整体 scale**。原因见下方 R1 专述 |

> **R1（可滚动网格的参照物问题）专述**：便单 4 象限不滚动，展开一个、另外 3 个始终在视口里，「缩放压暗整个网格」有参照物、联动成立。我们 7 格在可滚动网格里，**展开靠下的行时，前几行可能已滚出视口**——此时「缩放压暗一个已滚出的网格」几乎无视觉效果，联动落空，甚至「缩放一个只剩空白 `PAGE_BG` 的区域」毫无意义。所以背景联动改为：**① 全屏蒙层压暗**（无论展开哪行都成立、都有弹层浮起的纵深感）；**② 网格层仅轻微虚化/下沉，不缩放**（避免对可能已滚出的网格做无意义变换）。这一改避免了「展开靠下格子时背景无反应」的破绽。

### 4.7.4 实现方案

**(1) 统一外壳：抽出 `@Builder buildZoneShell(zone, color, content)`。**（评审采纳：用 Builder 而非共享常量）
外壳是「结构 + 视觉」一起复用——常量只能解决视觉参数（圆角/描边色），解决不了「白卡容器 + 标题行 + 内容插槽」这套**结构**。所以抽成 `@Builder buildZoneShell(zone, color, content)`：内部封装「白卡背景 + `CARD_RADIUS` 圆角 + `ZONE_*_STROKE` 描边 + 浅染标题行（色条/图标/名/进度，色条保 `TEXT_MAIN`）」，`content` 作为内容插槽。`ZoneGridCell`（收缩）和 `FocusedZoneView`（展开）**都调这个 Builder**，只是传入的 `content` 不同：收缩态传「4 件预览 + 「+N」」，展开态传「全量 List + 添加行」。这样外壳结构天然一致，是转场不闪烁的根基。

**(2) 展开态改为「卡片浮层」而非「整屏页」。**
`FocusedZoneView` 去掉 `backgroundColor(PAGE_BG)` 整屏底，改为：
- 外层卡片：`CARD_BG` + `CARD_RADIUS` + `ZONE_*_STROKE` 描边 + `SHADOW_MEDIUM` 阴影（浮起感）
- 尺寸：不再 `100%/100%`，改为「宽 = 屏宽 - 左右边距」、「高 = 内容自适应但封顶（避开顶部信息区/底部安全区），超出则卡片内 List 滚动」
- 定位：geometryTransition 自动负责从「原格子 Rect」插值到「展开 Rect」，不需要手动算坐标——**前提是两端节点都参与同一 id 的 geometryTransition 且外观一致**

**(3) 背景联动（全屏蒙层 + 网格虚化，不缩放网格）。**（评审采纳 R1）
展开瞬间叠一层**覆盖全屏的半透明压暗蒙层**（如 `#33000000`），与 `SPRING_HERO_EXPAND` 同帧淡入；收起时反向，与 `SPRING_HERO_COLLAPSE` 同帧淡出。蒙层覆盖全屏，**不依赖网格是否在视口**，所以无论展开哪一行都有稳定的「弹层浮起」纵深感。
- 网格层（`buildGridState`）只做**轻微 blur 虚化 + 极小 translateY 下沉**，**不做整体 scale**——因为 R1：网格可滚动，靠下行展开时前几行可能已滚出视口，对一个「可能已滚出/只剩空白」的网格做缩放无意义甚至穿帮。
- 可复用 Index.ets L1051-1069 的 `contentBlur` 思路（blur 部分），但**去掉 `contentScale` 缩放部分**，并把压暗从「依附网格」改为「独立全屏蒙层」。
- 蒙层 `onClick` → 触发收起（见下方第 5 点），符合「点弹层外关闭」直觉。

**(4) 进退场曲线保持现有 Hero token（已就绪，不新增）。**
- 展开：`SPRING_HERO_EXPAND`（0.42/0.73）
- 收起：`SPRING_HERO_COLLAPSE`（0.36/0.78）
- 关键：`focusedZone` 赋值 + `focusedZoneRender` 赋值必须在**同一个 animateTo 帧内**（现有 `openFocus` L236-241 已正确这么做），保证 geometryTransition 同帧拿到「id 匹配 + 源 Hidden / 目标 Visible」，否则退化为普通淡入淡出。

**(5) 收起交互保留。**
现有「点空白 / 左右划收起」（`FocusedZoneView` L183-193）保留。改为卡片浮层后，**点卡片外的压暗背景**也应收起（新增背景蒙层 onClick → onClose），更符合「点外面关弹层」直觉。

### 4.7.5 风险与验证点

- **geometryTransition 对「内容差异大」两端的表现**：HarmonyOS 的 geometryTransition 在两端尺寸/层级差异大时可能闪烁或配对失败。统一外壳（4.7.4-1）正是为降低这个风险——外观越一致，插值越顺。需真机验证转场无闪烁、无错位。
- **展开态卡片内 List 滚动 vs 外层手势冲突**：卡片内 List 纵向滚动、外层横划收起，需确认手势不打架（现有 `PanGesture Horizontal distance:24` 已做方向隔离，沿用）。
- **首屏 cellHeight 反算**：展开态不再铺满，但收缩态 cellHeight 逻辑不变，互不影响。
- **必须真机录屏对比便单**：转场是否「边框连续放大」、是否还有「翻页感」，以录屏逐帧核对为验收标准。

## 6. 分阶段落地排序（每步独立成立、可构建验证、即时 commit）

> 遵守开发铁律：最小改动、一次只做一件事、改完即 `hvigorw assembleApp` 验证、通过即 commit。

**Phase A — 拖动态去灰（最高优先级，最低成本，最高感知收益）**
- 仅改 `UnifiedChecklistView` 蒙层 dragging 淡出（§4.4.1）+ 胶囊浮起（§4.4.2）。
- 一处实现遗留，1 个文件，立竿见影，先把最丑的中间态干掉。

**Phase B — 容器化（解决「简陋」+「区分不清」两大主诉求）**
- 新增 `ZONE_*_STROKE` token。
- 满格子白卡容器化（§4.1）+ 空格子压缩淡化加提示（§4.2）+ 列高对齐（§4.3）。
- 这一步做完，收缩态质感跃迁，空/满层级正过来。

**Phase C — 拖动物理感深化**
- 源位虚线空槽（§4.4.3）+ 目标格高亮微调（§4.4.4）。

**Phase D — 展开态容器化 + 共享元素转场重构（§4.5 + §4.7，对标便单核心诉求）**（评审采纳：D 与 D+ 合并）
- 合并理由：转场重构（§4.7）的核心依赖「两端共用外壳」正是展开态容器化（§4.5）的产出。拆开会导致「先做一版展开容器、转场时再推翻重做」，是同一功能的两面，必须一次做对。
- 步骤：抽 `@Builder buildZoneShell` → `ZoneGridCell` 与 `FocusedZoneView` 都接外壳 → 展开态改卡片浮层（不铺满、目标 Rect 锚定原位中心）→ 全屏蒙层压暗 + 网格虚化下沉（不缩放）→ 真机录屏逐帧对比便单。
- 依赖 Phase B（外壳描边、标题行浅染先就位）；这是「边框自然放大」体感的关键一步，作为本轮重头戏单独验收。

**Phase E — 顶部信息区 + 进度条 + 勾选态（§4.6 + 勾选强化）**
- 锦上添花层，把「行程档案」的精致感拉满。

> 纪律：A→E 顺序执行，每个 Phase 单独构建验证 + commit。每步开始前对照本文档与 vision v3，确认不偏离「格子即清单」范式。
