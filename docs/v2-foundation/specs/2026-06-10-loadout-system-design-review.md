# 塔科夫式配装系统 Spec v1 · 审查意见

> 审查日期：2026-06-10 | 审查者：Everest  
> 审查对象：`docs/v2-foundation/specs/2026-06-10-loadout-system-design.md` (v1 草稿)  
> 审查标准：第一性原理 × 用户体验 × 项目架构一致性 × 向后兼容

---

## 0. 总体评价

这份 spec 的宏观方向正确——Zone 卡片网格、品类→槽位自动映射、零迁移策略、一期范围克制，都是合理的工程判断。骨架可以保留。

但在三个关键决策上存在**用户体验风险**，需要修正后才能进入开发。以下按优先级逐项展开。

---

## 🔴 P0 · 必须修正

### 1. 「完全替代 ChecklistDetail」决策过于激进（§1 决策 #1 + §9）

#### 1.1 问题描述

Spec 提出 LoadoutPage **完全替代** ChecklistDetail 作为行程详情页，并计划后续物理删除 ChecklistDetail.ets。

#### 1.2 为什么这是问题

当前 ChecklistDetail（~853 行）不是一个简单的 checkbox 列表。它承载了以下不可丢弃的功能：

| 功能 | 位置 | 说明 |
|------|------|------|
| **共享元素转场** | `geometryTransition('trip-' + id)` | HomePage 行程卡片 → ChecklistDetail 一镜到底展开。如果 LoadoutPage 替代它，这个转场必须完整移植——否则页面跳转从「共享元素展开」退化为「硬切」，这是用户能明显感知的体验退步 |
| **渐进式 chip 录入** | Header 中的 `profileSummary()` | v0.6.0 地基层核心交付。行程结构化字段（目的地/里程/海拔/爬升/时长）以一行摘要展示，点击进入 ProfileEditSheet 编辑 |
| **分组折叠/展开** | `GroupHeader` Builder + `onToggleGroup` | 按品类分组，用户可折叠不关注的分组。对于 30+ 装备的行程，这是信息密度管理的关键手段 |
| **左滑编辑/删除** | `swipeAction` Builder | 装备行左滑露出编辑/删除按钮 |
| **打勾进度动画** | 100% 完成庆祝动画 + `CompletionToast` | 仪式感体验 |
| **折叠头部** | `CollapsingHeader` + `CollapsedHeader` | 滚动驱动标题缩放/收起，节省纵向空间 |
| **核查复盘入口** | `ActionRow` Builder → `openReviewMode()` | 逐张卡片滑动确认装备是否带齐 |

Spec 的「功能对齐清单」§9.2 直接声明 **分组折叠/展开 → ❌ 移除**。对于 30+ 装备的行程，所有装备平铺在 Zone 卡片中——用户需要滚动很久才能看完。

#### 1.3 第一性原理分析

**行程详情的本质是什么？** 用户打开一个行程，有两种核心意图：

- **出行前**（packing 时态）：我想*规划*带什么——结构化思考、查漏补缺。配装视图是这时的正确工具。
- **出行中/出行后**（checking 时态）：我想*确认*带了没有——快速扫视、逐项打勾。清单视图是这时的正确工具。

两种意图对应两种心智模型，强行合并会导致「开 SUV 去买菜」——工具很强大但不顺手。

#### 1.4 修正方案

**不要「完全替代」，改为「并存 + 渐进迁移」。**

```
行程详情页 = 顶部双模式切换

┌──────────────────────────────────────┐
│  < 行程名称                    ···    │
│                                      │
│  [配装视图]  [清单视图]               │  ← 两个 Tab/分段
│                                      │
│  ┌──────────┐  ┌──────────┐         │
│  │  上身     │  │  头部     │         │  ← 配装视图内容
│  │  …       │  │  …       │         │
│  └──────────┘  └──────────┘         │
└──────────────────────────────────────┘
```

**具体实现：**

1. 行程详情页改为一个 `NavDestination`，内部有两个子视图，通过顶部分段控件切换：
   - **配装视图（默认）**：新增的 LoadoutPage（Zone 卡片网格）
   - **清单视图**：现有的 ChecklistDetail（保留全部功能，仅作为第二个 tab）
2. `geometryTransition` 的 id 从 `'trip-' + id` 保持不变——共享元素转场继续工作，目标组件改为新的父 NavDestination
3. 开发期间两个视图并存。发布后收集使用反馈：
   - 如果用户 80% 时间在配装视图，清单视图可以后续简化（去掉折叠头部等）
   - 如果用户出行当天频繁切换回清单视图，说明两种视图各有价值，不应合并
4. ChecklistDetail 的代码**不删除**，仅将其作为子组件嵌入新页面。如果未来确实不再需要，再物理删除

**数据共享**：两个视图读取同一份 `TripChecklist` 数据。在配装视图中修改装备（添加/删除/打勾），清单视图实时同步（因为共享同一个 `checklists` @State 引用，Index.ets 的 `applyChecklistState()` 机制保证了这一点）。

### 2. Misc Zone 是设计黑洞（§2.1 + §3.2）

#### 2.1 问题描述

Spec 将 13 个品类中的 **7 个** 全部映射到 Misc Zone：证件、饮食、电力、摄影、急救、洗护、其他。Misc 卡片 span 全宽，其他 6 个 Zone 各占半宽。

#### 2.2 为什么这是问题

对于典型户外用户的真实装备分布：

| Zone | 典型装备数量 | 卡片尺寸 |
|------|------------|---------|
| Head | 1-3 件（帽子、墨镜、头灯） | 半宽，大量空白 |
| UpperBody | 3-5 件（速干、抓绒、硬壳、羽绒） | 半宽，适中 |
| LowerBody | 1-3 件（内裤、登山裤、雪裤） | 半宽，少量空白 |
| Feet | 1-2 件（袜子、登山鞋） | 半宽，大量空白 |
| Carry | 1-3 件（主包、腰包、冲顶包） | 半宽，少量空白 |
| Sleep | 2-4 件（睡袋、地垫、帐篷） | 半宽，适中 |
| **Misc** | **12-26 件**（证件+炉头+气罐+锅+餐具+水袋+充电宝+数据线+相机+急救包+药品+牙刷牙膏+毛巾+防晒霜+其他） | 全宽，**严重超载** |

视觉结果：页面上方 6 个清爽的小卡片（有些几乎是空的），下方一个巨大的杂物抽屉。这不仅视觉失衡，更重要的是——**用户想找「充电宝在哪」时，需要在 Misc 的 20 件装备中翻找，这和旧版清单的问题完全一样。**

#### 2.3 第一性原理分析

「结构强度光谱」（纲领 §4.6）的核心思想是：**不同类别的装备有不同的结构强度，UI 形式应随结构强度递减**：

- 衣物：结构最强 → 槽位（按身体部位 × 分层）
- 睡眠：结构次之 → 堆栈（地垫→睡袋→内胆→庇护所）
- 饮食：结构更弱 → 分组列表
- 零散品：结构最弱 → 朴素列表

将 7 个品类无差别地塞进 Misc，本质上是在 Misc 内部放弃了结构强度光谱——把饮食系统和洗护用品放在同一个无差别列表里，和旧版清单一样，没有「结构化」的优势。

#### 2.4 修正方案

**方案 A（推荐）：Misc 卡片内部按品类做子分区**

```
┌──────────────────────────────────────┐
│  杂项                                  │
│                                        │
│  ▸ 证件                                │  ← 子分区标题（可折叠）
│    身份证  ·  钱包                     │
│                                        │
│  ▸ 饮食系统                            │
│    炉头 · 气罐 · 钛锅 · 餐具           │
│                                        │
│  ▸ 电力系统                            │
│    充电宝 20000mAh · 数据线            │
│                                        │
│  ▸ 摄影 · 急救 · 洗护 · 其他           │
│    …                                   │
└──────────────────────────────────────┘
```

每个子分区是一个迷你 section：
- 标题行：品类名 + 装备数量
- 内容行：该品类下的装备（不超过 5 行）
- 可折叠（默认展开，但如果超过 4 个品类可考虑折叠低频品类）

这样做的好处：
- 保持了「一个 Misc 卡片」的简洁布局
- 内部按品类组织，用户查找时的心智模型是「饮食 → 炉头」，而非「杂项 → 翻找 20 件」
- 实现成本低——本质上就是 Misc 卡片内的 `ForEach` 两层嵌套（外层品类，内层装备）

**方案 B（如果 Misc 仍然太长）：拆分出「饮食」为独立 Zone**

纲领 §4.6 已经指出饮食系统的特殊性：「结构强度弱、但装备数量多」。可考虑将 `饮食系统` 从 Misc 中提出，作为独立的「厨房」Zone。这样 Misc 减到 6 个品类，饮食 Zone 有 3-5 件装备，各自合理。

如果选方案 B，需要更新 `GearLoadout.ets` 的 `CATEGORY_SLOT_MAP` 和 `BodyZone` 枚举。由于 `BodyZone` 是 enum，新增一个值不影响现有映射。但如果用户已有「饮食系统」的装备在 Misc 中，需要做运行时推断。

**两种方案可以结合**：一期做方案 A（Misc 子分区），如果实机测试发现 Misc 仍然太长，二期再做方案 B（拆分饮食 Zone）。

### 3. 品类锁死 13 项破坏现有功能（§1 决策 #7）

#### 3.1 问题描述

Spec 决策 #7：「锁死 DEFAULT_CATEGORIES（13 项），不支持用户自建品类」。

理由：「品类与槽位映射强绑定，松绑会破坏自动落槽」。

#### 3.2 为什么这是问题

当前系统的品类管理功能是**完整且经过验证的**：

| 能力 | 代码位置 | 状态 |
|------|---------|------|
| 新增品类 | `CategoryService.addCategory()` | ✅ 已实现并持久化 |
| 重命名品类 | `CategoryService.renameCategory()` | ✅ v0.5.9 修复了 rename bug |
| 删除品类 | `CategoryService.deleteCategory()` | ✅ 含装备迁移 |
| 拖拽排序 | `CategoryService.reorderCategories()` | ✅ `categoryOrder` 持久化 |
| 品类筛选 | `CategoryTagGroup` + multi-select | ✅ 全功能 |
| UI 增删改 | `CategoryTagGroup` 的编辑模式（长按进入） | ✅ 含抖动动画 |

如果锁死品类：
1. **现有用户的自定义品类装备全部找不到 Zone 归属**（虽然 `slotHintForCategory` 有 fallback，但用户体验是「我的滑雪装备去哪了？」）
2. **用户无法新增场景品类**——「滑雪」「攀岩」「溯溪」「骑行」每个场景都有不同装备组合，用户需要按场景管理
3. **`CategoryTagGroup` 的编辑模式全部变成死代码**

#### 3.3 为什么锁死不必要

`slotHintForCategory()` 已经有完善的 fallback 机制（`GearLoadout.ets:47-53`）：

```typescript
export function slotHintForCategory(category: string): SlotHint {
  const hint = CATEGORY_SLOT_MAP[category];
  if (hint !== undefined) {
    return hint;
  }
  // 未知品类 → 回退到 Misc/Accessory
  const fallback: SlotHint = { zone: BodyZone.Misc, layer: LayerOrder.Accessory };
  return fallback;
}
```

任何不在 13 个预设中的品类，自动落入 Misc。**这正是为此场景设计的**——用户在品类上的自由度不需要牺牲，未知品类有一个合理的兜底。

#### 3.4 修正方案

**保留现有品类扩展能力，不锁死。**

1. 用户仍可创建/重命名/删除自定义品类（`CategoryTagGroup` 编辑模式保持不变）
2. 预设的 13 个品类按 `CATEGORY_SLOT_MAP` 精确映射
3. 自定义品类自动 fallback 到 Misc/Accessory（已有代码处理）
4. **可选增强（非一期）**：在装备编辑页新增一个可选字段「配装区域」，用户可以手动覆盖某件装备的自动映射结果。这个字段存到 `ChecklistItem.group` 中，不影响 `GearItem` 的数据模型。默认值是「自动」，代表使用 `CATEGORY_SLOT_MAP` 推断。

这样既不增加录入负担（品类照常填，大多数情况自动映射正确），又给了高级用户手动微调的能力。

## 🟡 P1 · 建议修正

### 4. GearPickerSheet 缺少目标预览（§3.2）

#### 4.1 问题描述

当前流程：用户在 Sheet 中 tap 装备 → Sheet 关闭 → 装备「凭空」出现在某个 Zone 卡片中。用户选装备时不知道它会落到哪个区域。

#### 4.2 为什么这是问题

第一次使用的用户会困惑——「我刚选的炉头去哪了？」需要在 7 个 Zone 卡片中找刚才选的东西。这是「自动落槽」的黑暗面：自动化程度越高，用户对结果的控制感和可预测性越低。

#### 4.3 修正方案

**在 GearPickerSheet 的每行装备右侧显示目标 Zone 标签，同时 Sheet 选中装备后不自动关闭。**

```
┌──────────────────────────────────────┐
│  GearPickerSheet          [关闭]      │
│  ┌────────────────────────────────┐  │
│  │  🔍 搜索装备…                   │  │
│  └────────────────────────────────┘  │
│  证件  穿着·上身  饮食  …            │  ← CategoryTagGroup 品类筛选
│                                        │
│  ▸ 穿着·上身                           │
│  ┌──────────────────────────────────┐  │
│  │  速干T恤 (Patagonia)    → 上身   │  │  ← 目标 Zone chip
│  │  抓绒衣 (Arc'teryx)  ✓  → 上身   │  │  ← 已选标记
│  │  硬壳冲锋衣 (Arc'teryx) → 上身   │  │
│  └──────────────────────────────────┘  │
│  ▸ 饮食系统                            │
│  ┌──────────────────────────────────┐  │
│  │  炉头 (Soto)           → 杂项   │  │
│  │  钛锅 (Snow Peak)      → 杂项   │  │
│  └──────────────────────────────────┘  │
└──────────────────────────────────────┘
```

**关键交互规则**：

1. GearPickerSheet **不自动关闭**（spec 已提到「支持连续多选模式」——保持这点）
2. 选中装备后，背景 LoadoutPage 中对应的 Zone 卡片出现装备入场动画（scale + fadeIn），用户能**同时看到**选中反馈和落点
3. 用户可连续选择多件装备，最后手动关闭 Sheet（下滑或 tap 关闭按钮）
4. 已在当前行程中的装备显示 ✓ 标记，不可重复添加（或 tap 后取消）

这实现了「所见即所得」——用户在选择时就知道每件装备会落到哪里。

### 5. 退出配装页的路径未定义（§3.1）

#### 5.1 问题描述

Spec 定义了两个入口（新建行程 → 进入、行程列表 tap → 进入），但未定义如何退出。

#### 5.2 修正方案

```
LoadoutPage 顶部导航栏：

┌──────────────────────────────────────┐
│  < 返回    武功山徒步         ···    │  ← 左侧返回箭头 + 行程标题
│                                      │
│  已装包 12/18  ████████░░░  67%     │  ← 进度条
└──────────────────────────────────────┘
```

- **返回箭头** `<`：点击返回行程列表（HomePage）。返回时自动保存当前配装结果——不需要显式「保存」按钮。
- **不保存退出**：如果用户从进入配装页到退出之间没有任何变更（未添加/删除/勾选装备），返回时不触发 `saveChecklists()`，避免不必要的 I/O。
- **更多菜单** `···`：右上角，包含「编辑行程信息」「删除行程」等操作（对齐 ChecklistDetail 现有功能）。

### 6. 「层级微调」的交互未定义（§2.2 + §8.2）

#### 6.1 问题描述

Spec §2.2 提到「用户配装时可在卡片内微调层级（tap 层级 chip 切换）」，但 §8.2 又说「一期提供 long-press → 菜单中的移动到其他区域选项」。层级微调和区域移动是两个不同的操作，但交互方式都指向 long-press——这会造成歧义。

#### 6.2 修正方案

**明确两个操作的交互方式**：

| 操作 | 触发方式 | 效果 |
|------|---------|------|
| 切换勾选 | tap 装备行 | checked 切换（最常用，最高优先级） |
| 修改层级 | tap 装备行左侧的层级 badge | 弹出层级选择 popup（Base/Mid/Shell/Insulation/Accessory） |
| 移动到其他 Zone | long-press → 菜单 →「移动到」→ 选择目标 Zone | 修改 `ChecklistItem.group` |
| 查看装备详情 | long-press → 菜单 →「查看详情」 | 呼出装备详情面板 |
| 从行程移除 | long-press → 菜单 →「移除」 | 从 `items` 中删除该装备 |

层级 badge 作为独立的 tap target，不与装备行 tap 冲突。

### 8. ReviewPage 的路径需要更新

当前 `ReviewPage.ets`（核查复盘——逐张卡片滑动确认）从 ChecklistDetail 的 `ActionRow` 按钮进入。如果 LoadoutPage 替代/并存 ChecklistDetail，需要明确：

- 配装视图是否也有「核查复盘」入口？（建议有——配装完后一键进入核查模式）
- 核查复盘退出后回到配装视图还是清单视图？（建议回到进入时的视图）

### 10. 配装视图的底部按钮应该是 FAB

当前 ChecklistDetail 的「添加装备」按钮在底部操作栏中。按交互入口原则，配装视图的「＋ 添加装备」应该是 FAB（浮动操作按钮）——右下角悬浮，内容可穿透，与装备库 Tab 的 FAB 保持一致的位置和交互模式。

### 11. 2 列网格缺少视觉层次提示

Spec 使用纯卡片网格，没有人体拓扑线条连接 Zone。建议在 Zone 卡片标题旁加微小的 `sys.symbol` 图标作为视觉线索：

| Zone | 建议 icon |
|------|----------|
| Head | `sys.symbol.face.smiling` |
| UpperBody | `sys.symbol.tshirt` |
| LowerBody | `sys.symbol.figure.walk` |
| Feet | `sys.symbol.shoeprints.fill` |
| Carry | `sys.symbol.backpack` |
| Sleep | `sys.symbol.tent` |
| Misc | `sys.symbol.square.grid.2x2` |

这些 icon 给卡片增加辨识度，用户扫一眼就知道哪个卡片对应哪个部位，不需要读完标题文字。
